Star Schema - Comprehensive Notes

Definition:

A star schema is the simplest type of data warehouse schema consisting of:

  • One central fact table (contains measurements/metrics)
  • Multiple dimension tables (contain descriptive attributes) connected directly to the fact table

Characteristics:

  • Resembles a star with the fact table at the center
  • Denormalized structure (dimension tables contain redundant data)
  • Simple queries with fast performance
  • Easy to understand and maintain

Example: Retail Sales Data Warehouse

Fact_Sales (fact table)
- sale_id (PK)
- product_key (FK)
- customer_key (FK)
- store_key (FK)
- time_key (FK)
- quantity_sold
- sale_amount

Dim_Product (dimension table)
- product_key (PK)
- product_name
- category
- price
- brand

Dim_Customer (dimension table)
- customer_key (PK)
- customer_name
- address
- phone
- email

Dim_Store (dimension table)
- store_key (PK)
- store_name
- location
- manager
- size_sqft

Dim_Time (dimension table)
- time_key (PK)
- date
- day_of_week
- month
- quarter
- year

Derived From:

  • Developed from the dimensional modeling approach by Ralph Kimball
  • Evolved from early decision support systems in the 1980s
  • Commonly used in data marts and departmental data warehouses

When to Use:

  • You need simple, fast queries for reporting
  • Your dimensions aren't overly complex
  • Disk space isn't a major concern
  • You're building a departmental data mart

Snowflake Schema - Comprehensive Notes

Definition:

A snowflake schema is a variation of the star schema where:

  • Dimension tables are normalized into multiple related tables
  • Forms a shape resembling a snowflake

Characteristics:

  • Normalized structure (reduces data redundancy)
  • More complex queries due to joins
  • Saves storage space
  • Better for complex hierarchical dimensions

Example: Extended Retail Sales (Snowflaked)

Fact_Sales (same as star schema)

Dim_Product
- product_key (PK)
- product_name
- category_key (FK)  -- Now references a category table
- price
- brand_key (FK)    -- Now references a brand table

Dim_Product_Category
- category_key (PK)
- category_name
- department_key (FK)

Dim_Department
- department_key (PK)
- department_name

Dim_Product_Brand
- brand_key (PK)
- brand_name
- manufacturer_key (FK)

Dim_Manufacturer
- manufacturer_key (PK)
- manufacturer_name
- country

Derived From:

  • Extension of the star schema concept
  • Developed to handle more complex business hierarchies
  • Commonly used when storage optimization is critical
  • Often seen in enterprise data warehouses

When to Use:

  • You have complex hierarchical dimensions
  • Storage optimization is important
  • You're building an enterprise data warehouse
  • Your dimensions have many attributes that could be normalized

Comparison with Star Schema

Feature Star Schema Snowflake Schema
Structure Denormalized Normalized
Query Performance Faster (fewer joins) Slower (more joins)
Storage Uses more space More space-efficient
Complexity Simpler More complex
Maintenance Easier Harder
Typical Use Data marts, simple analytics Enterprise DW, complex hierarchies

Data Models - Comprehensive Notes

Definition:

Data models are abstract models that organize elements of data and standardize how they relate to one another and to properties of real-world entities.

Types of Data Models:

  1. Conceptual Data Model - High-level, business-oriented
  2. Logical Data Model - Detailed, technology-independent
  3. Physical Data Model - Implementation-specific

Common Data Modeling Techniques:

  • Entity-Relationship (ER) Modeling
  • Dimensional Modeling (used in data warehouses)
  • Object-Oriented Modeling
  • Hierarchical Modeling
  • Network Modeling

Example: ER Diagram Components

Entities:    [Student]       [Course]
Attributes:  student_id      course_id
             student_name    course_name
             email           credit_hours

Relationships:
Student --<Enrolls>-- Course

Data Modeling Process:

  1. Requirement analysis
  2. Conceptual design
  3. Logical design
  4. Physical design
  5. Implementation
  6. Maintenance and refinement

Best Practices:

  • Start with clear business requirements
  • Use appropriate naming conventions
  • Document assumptions and decisions
  • Validate with stakeholders
  • Consider future growth

Object-Oriented Programming in Databases

Definition:

Object-oriented programming (OOP) in databases combines database capabilities with object-oriented programming principles.

Key Concepts:

  • Objects: Represent real-world entities with attributes (data) and methods (behavior)
  • Classes: Blueprints for creating objects
  • Inheritance: Classes can inherit properties from other classes
  • Polymorphism: Objects can take many forms
  • Encapsulation: Bundling data with methods that operate on that data

Database Applications:

  • Object-Relational Mapping (ORM): Technique to convert data between OOP and relational databases
  • Object-Oriented Databases: Store objects directly (e.g., MongoDB, CouchDB)

Example: Python ORM with SQLAlchemy

from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.ext.declarative import declarative_base

Base = declarative_base()

class Customer(Base):
    __tablename__ = 'customers'
    
    id = Column(Integer, primary_key=True)
    name = Column(String)
    email = Column(String)
    
    def __repr__(self):
        return f"<Customer(name='{self.name}', email='{self.email}')>"

# Create database connection
engine = create_engine('sqlite:///mydatabase.db')
Base.metadata.create_all(engine)

Advantages:

  • More intuitive modeling of complex relationships
  • Better code organization and reusability
  • Smoother integration with application code

Database Backup Strategies

Definition:

Database backup is the process of creating copies of database contents to protect against data loss.

Backup Types:

  • Full Backup: Complete copy of the entire database
  • Incremental Backup: Only changes since last backup
  • Differential Backup: All changes since last full backup
  • Transaction Log Backup: Backup of transaction logs

Backup Methods:

  • Physical Backup: Copy of database files
  • Logical Backup: Export of database objects
  • Hot vs. Cold Backup: With/without database shutdown

Example: MySQL Backup Command

# Full database backup
mysqldump -u username -p database_name > backup.sql

# Restore from backup
mysql -u username -p database_name < backup.sql

# Automated backup script example
#!/bin/bash
mysqldump -u root -pPassword123 my_database | gzip > /backups/db_$(date +%Y%m%d).sql.gz

Best Practices:

  • Follow the 3-2-1 rule (3 copies, 2 media types, 1 offsite)
  • Test restore procedures regularly
  • Automate backup processes
  • Encrypt sensitive backup data

Database Privileges and Security

Definition:

Database privileges control what actions users can perform on database objects.

Common Privileges:

  • System Privileges: Database-wide actions (CREATE USER, CREATE TABLE)
  • Object Privileges: Actions on specific objects (SELECT, INSERT, UPDATE)
  • Role-Based Privileges: Assign privileges to roles, then roles to users

Security Principles:

  • Least Privilege: Users get only necessary permissions
  • Separation of Duties: Critical tasks require multiple users
  • Defense in Depth: Multiple security layers

Example: PostgreSQL Privileges

-- Create user
CREATE USER analyst WITH PASSWORD 'secure123';

-- Grant SELECT on specific table
GRANT SELECT ON employees TO analyst;

-- Create role with privileges
CREATE ROLE reporting;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO reporting;

-- Assign role to user
GRANT reporting TO analyst;

-- Revoke privileges
REVOKE INSERT ON customers FROM analyst;

Security Best Practices:

  • Regularly audit user privileges
  • Use strong password policies
  • Implement row-level security when needed
  • Encrypt sensitive data at rest and in transit

Database Views - Comprehensive Notes

Definition:

A view is a virtual table based on the result-set of an SQL statement. It contains rows and columns, just like a real table.

Advanced View Concepts:

  • Materialized Views: Physically store query results for faster access
  • Updatable Views: Views that allow INSERT/UPDATE/DELETE operations
  • Indexed Views: Materialized views with indexes for performance

View Optimization Techniques:

  • Use WHERE clauses in view definitions to filter early
  • Limit the number of columns in views
  • Consider materialized views for complex aggregations

Example: Oracle Materialized View

CREATE MATERIALIZED VIEW sales_summary
REFRESH COMPLETE ON DEMAND
AS 
SELECT product_id, 
       SUM(quantity) as total_quantity,
       SUM(amount) as total_sales
FROM sales
GROUP BY product_id;

-- Create index on materialized view
CREATE INDEX idx_sales_summary ON sales_summary(product_id);

View Security Considerations:

  • Views can expose only necessary columns to users
  • Implement row-level security through views
  • Use WITH CHECK OPTION to validate updates

Database Query Optimization

Query Processing Stages:

  1. Parsing: Check SQL syntax
  2. Optimization: Determine most efficient execution plan
  3. Execution: Run the optimized plan

Optimization Techniques:

  • Indexing: Proper indexes on frequently queried columns
  • Query Rewriting: Restructure queries for better performance
  • Join Optimization: Choose optimal join algorithms
  • Partitioning: Divide large tables into manageable pieces

Example: SQL Server Query Plan

-- Analyze query execution plan
SET SHOWPLAN_TEXT ON;
GO
SELECT c.CustomerName, o.OrderDate, o.TotalAmount
FROM Customers c
JOIN Orders o ON c.CustomerID = o.CustomerID
WHERE o.OrderDate > '2023-01-01';
GO
SET SHOWPLAN_TEXT OFF;

-- Create covering index
CREATE INDEX idx_orders_customer_date
ON Orders(CustomerID, OrderDate)
INCLUDE (TotalAmount);

Query Performance Best Practices:

  • Avoid SELECT * - specify only needed columns
  • Use parameterized queries to prevent SQL injection
  • Limit result sets with WHERE clauses
  • Consider stored procedures for complex queries
